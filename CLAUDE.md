## 1. Первым делом — про дизайн

**Дизайн-система: `design-system.md` — единственный источник правды по визуалу.**

Перед любой UI-задачей (новая страница, ремонт старой, добавление компонента) — прочитать `design-system.md` в корне проекта. Там полный набор: философия, токены, компоненты, per-tool identity, чек-лист приёмки.

**Важно:** любые упоминания дизайна в старом коде, комментариях или других MD-файлах, которые противоречат `design-system.md`, считаются legacy и подлежат замене. Если встретил, например, `--grad-btn` в CSS, `linear-gradient` на заголовке или Syne в подключении шрифтов — это старая система, её нужно заменить по правилам из раздела 11 «Legacy — что удаляется/заменяется» в `design-system.md`.

Файл `SYSTEM.md` (если ещё лежит в корне) — legacy. Игнорировать, использовать `design-system.md`.

## 2. Стек

**Бэкенд:** Python 3.12, FastAPI + Uvicorn, Jinja2 (HTML-шаблоны), SQLAlchemy + SQLite, bcrypt + python-jose, httpx для внешних API.

**Фронтенд:** Ванильный JS + чистый CSS в шаблонах. Никакого React, Node.js, сборщиков. Это не SPA.

**Иконки:** Lucide через CDN (`https://unpkg.com/lucide@latest`). Эмодзи в UI-элементах запрещены — см. `design-system.md`, раздел 8.

**ИИ (через OpenRouter API):**
- `LETTER_MODEL` (по умолчанию `anthropic/claude-opus-4-8`) — генерация сопроводительных писем
- `ANALYZE_MODEL` (по умолчанию `anthropic/claude-sonnet-4-5`) — анализ вакансий, оценка релевантности
- `PARSER_MODEL` (по умолчанию `anthropic/claude-sonnet-4-5`) — парсинг резюме в досье
- `anthropic/claude-haiku-4-5` — лёгкие задачи (нутриция, простые классификации)

**Деплой:** Fly.io, автодеплой через GitHub Actions на push в `main`.

## 5. Важные архитектурные нюансы

- **Резюме пользователя** живёт в HH-ассистенте (`/hh` → вкладка «Резюме»), НЕ в профиле (`/profile`). Это архитектурное решение: резюме — данные для HH-модуля, а не общий профильный атрибут.
- **Профиль** содержит: аватар, отображаемое имя, email + смена email, смена пароля, часовой пояс, уведомления, удаление аккаунта. См. `design-system.md`, раздел 10.1.
- **Аватар пользователя** отображается в верхней шапке справа (кругляшка 32×32). Клик открывает dropdown с ссылками (Профиль, Админ, Выйти). Отдельные текстовые ссылки в шапке НЕ используются.
- **Sticky-навигация** — верхняя шапка всегда прикреплена при скролле. См. `design-system.md`, раздел 4.
- **Глобальный поиск** в шапке — компонент заложен в UI, реализация как отдельная задача. См. раздел 5.
- **TRAINER_PROMPT.md** исключён из `.dockerignore` намеренно — он нужен в проде.
- **SQLite база** — `app.db`, volume `/data` на Fly.io.

## 6. Правила работы с кодом

- Разбивать работу по фазам, отчёт после каждой, всегда показывать реальный код (не summary).
- Перед добавлением UI-элемента — проверить, есть ли уже такой в `design-system.md`, раздел 2 «Компонентная база». Если есть — использовать существующий класс. Если нет — сначала обсудить, что за компонент, и добавить в дизайн-систему.
- CSS-переменные — только из `design-system.md`, раздел 13. Никаких новых токенов без обсуждения.
- Хардкод-цвета в HTML/CSS — запрещены. Всё через переменные.
- Хардкод-размеры шрифтов и отступов — запрещены. Всё через шкалу (`--text-*`, `--space-*`) или `clamp()`.
- Inline `<style>` в шаблонах Jinja2 — по возможности избегать. Общее — в `style.css`. Если нужен uniquely-page стиль — оформлять в отдельный `<style>`-блок в начале шаблона с комментарием «TODO: перенести в модульный CSS».

## 7. Известные технические долги

- В `hh.html` (~1940 строк) сейчас 165 хардкод-цветов и 143 уникальных класса — большая часть в inline `<style>` в шапке. Приоритетный кандидат на рефакторинг в рамках дизайн-миграции.
- Программа тренировок (`workout`) — не допилена: подбор упражнений через LLM без чёткой системы разминки/заминки, импорт видео из YouTube в процессе (см. `YOUTUBE_IMPORT_PROGRESS.md`).
- Дневник питания — вертикальный sidebar на десктопе (mobile-first порт, требует переработки под горизонтальный layout).
- Аватар пользователя как feature — не реализован (сейчас в шапке текстовые ссылки).
- Глобальный поиск как feature — не реализован (сейчас нет).
- Смена email как feature — не реализована.
- Удаление аккаунта как feature — не реализовано.

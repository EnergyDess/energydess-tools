# fable-brief.md — Эталонная сессия дизайн-миграции

> **Для кого:** Claude Fable 5 в Claude Code
> **Что делать:** переработать energydess.ru по новой дизайн-системе. HH-ассистент как эталон, остальные инструменты — в следующих сессиях.
> **Сколько времени:** одна длинная сессия. Не разбивать на маленькие подзадачи — держи весь контекст в голове.

---

## Задача

Переработать визуальный слой сайта energydess.ru согласно новой дизайн-системе. Это не косметика — это архитектурная миграция: удаление legacy-эффектов, унификация токенов, замена шрифтов, переработка компонентной базы.

**Что тебе НЕ надо трогать:**
- `main.py` — все эндпоинты, роуты, бизнес-логика, работа с БД. Всё это работает и правильно.
- `database.py`, `auth.py`, `zepp_client.py`, `exercises_data.json` — не трогаем.
- `TRAINER_PROMPT.md`, `YOUTUBE_IMPORT_PROGRESS.md` — не трогаем.

**Что ты трогаешь:**
- `static/style.css` — переработка целиком под новую токенную систему
- `templates/*.html` — приведение к дизайн-системе, приоритет: `hh.html` (эталон)
- `static/effects.js`, `static/theme.js` — удаление
- `SYSTEM.md` — удалить или переименовать в `SYSTEM.md.legacy`
- Добавить `static/noise.svg` — тонкий SVG-noise для atmospheric-слоя лендинга
- Добавить partial-шаблоны (см. фазу 3)

## Фазы работы

### Фаза 1. Фундамент — `style.css`

**Цель:** новая токенная система + компонентная база.

1. Полностью переписать `:root` в `style.css` по разделу 13 из `design-system.md`.
2. Удалить блок `html.light { ... }` целиком (мёртвый код).
3. Удалить все legacy-переменные: `--grad-btn`, `--font-head`, `--bg4`, старые `--accent`/`--accent-2` (заменяются на новую систему).
4. Обновить подключение шрифтов в HTML `<head>` — убрать Syne и DM Mono, добавить Manrope (все веса) и JetBrains Mono. Обновить в общем layout-шаблоне (если есть `base.html`) или во всех страницах.
5. Добавить в `style.css` компонентную базу: `.card`, `.btn-primary`, `.btn-secondary`, `.btn-ghost`, `.btn-signature`, `.input`, `.textarea`, `.chip`, `.eyebrow`, `.modal`, `.divider`, `.site-header`, `.site-footer`, `.avatar`, `.tool-icon`, `.breadcrumb`. Каждый — по спецификации из `design-system.md`, раздел 2.
6. Реализовать running-border анимацию для `.btn-signature` (раздел 3 в design-system.md). При инициализации кнопки JS вычисляет периметр через `getTotalLength()` и записывает в CSS-переменную на элементе.
7. Добавить `@media (prefers-reduced-motion: reduce)` в конце файла.

**Отчёт после фазы 1:** покажи полный `:root` и все компонентные классы. Не summary — полный код.

### Фаза 3. Общий layout — partial-шаблоны

Создай (если ещё нет) переиспользуемые фрагменты. Jinja2 их поддерживает через `{% include %}`.

1. **`templates/_header.html`** — sticky-шапка сайта (раздел 4 design-system.md):
   - Логотип слева
   - Глобальный поиск в центре (заглушка UI, функциональность — TODO с комментарием в коде)
   - Аватар справа с dropdown (Профиль, Админ если admin, Выйти)
   - Backdrop-blur, sticky.
2. **`templates/_footer.html`** — полноценный футер (раздел 6 design-system.md).
3. **`templates/_avatar.html`** — компонент аватара с инициалами по умолчанию.

Вставить `{% include '_header.html' %}` и `{% include '_footer.html' %}` во все страницы, кроме auth-страниц (регистрация, вход) — там только логотип в шапке.

**Отчёт после фазы 3:** покажи `_header.html` и `_footer.html` целиком, и пример вставки в `index.html`.

### Фаза 5. HH-ассистент — эталон переработки

**Это самая большая часть сессии.** Полная переработка `templates/hh.html` (1940 строк).

**Что сделать:**

1. **Удалить всё inline `<style>` в шапке шаблона.** Всё общее — в `style.css` через переиспользуемые классы. Уникальные для HH стили — в новый файл `static/hh.css` (подключить только на этой странице).
2. **Заменить все hardcode-цвета** на переменные. `grep -oE "#[0-9a-fA-F]{3,6}|rgba?\(" hh.html` в конце должен дать 0.
3. **Класс `.theme-hh` на body** страницы.
4. **Добавить breadcrumb** в верхней части: `Инструменты / HH-ассистент`.
5. **Реорганизовать в табы:**
   - **Написать письмо** (default) — текущее содержимое (форма вакансии + custom_context + анализ + результат)
   - **История писем** — то, что сейчас снизу
   - **Досье** — то, что сейчас в свёрнутой карточке `.ds` / `.ds-toggle`
   - **Резюме** — переехало из профиля (создать новую вкладку, скопировать textarea + upload PDF из профиля)
   - Табы sticky под шапкой, оформление: Lucide-иконки + текст, активный таб — синяя линия снизу (`--accent-hh`).
6. **Заменить все `.dosie-*` стили** на переиспользуемые компоненты (`.card`, `.input`, `.textarea`, `.chip`) с локальными модификаторами где нужно.
7. **Кнопки:**
   - Primary «Сгенерировать письмо» — `.btn-primary` с `--tool-accent`
   - Secondary «Проверить совпадение с резюме» — `.btn-secondary`
   - Ссылка/tertiary «Уже есть аккаунт» и подобные — `.btn-signature` с running border
8. **Chips custom_context** — новый стиль `.chip` с активным состоянием.
9. **Alert `.alert-error`** — новый стиль с `--error` цветом, Lucide-иконка `AlertCircle`.
10. **Analysis-card** (результат анализа вакансии) — переработать в `.card` с внутренней структурой: score badge (крупная цифра из `--font-mono`), список совпадений/пробелов через `.chip`, reason как body-text.
11. **История писем внизу** — переработать в компактный список карточек с ленивой подгрузкой (сохранить существующую логику JS).
12. **Empty states:** «Ещё не написано ни одного письма. Начните с вакансии →» вместо «Здесь пусто».

**Что НЕ трогать в HH:**
- JS-логику (все fetch к API, обработка ответов, автосохранение draft в localStorage).
- Структуру данных досье (7 разделов).
- Логику двухэтапной генерации (analyze → generate).

**Отчёт после фазы 5:** покажи новый `hh.html` целиком + `hh.css` целиком. Скриншот страницы в 3-х состояниях: пустая (без вакансии), после анализа (score виден), с готовым письмом.

## Как проверять качество на ходу

Каждые 20-30 минут работы делай checkpoint:
1. Открой одну переработанную страницу в браузере.
2. Открой скриншот референса рядом (Linear, Vercel, Raycast — на твой выбор).
3. Спроси себя: «Это похоже на них по восприятию, или это ещё чувствуется как “твой сайт со старой основой”?» Если чувствуется — что конкретно выдаёт? Fix и продолжай.

Не полагайся только на код-проверки. Визуальная согласованность — главное.
